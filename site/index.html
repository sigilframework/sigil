AI is consuming enormous resources. Sigil is built from the ground up to experiment on long-running agents with less token and energy usage.

🌍 Why

AI has a resource problem

Every time an AI agent answers a question, writes code, or runs a task, it spins up fresh infrastructure, loads context from scratch, and throws everything away when it's done. It's like starting your car engine from cold every time you want to drive a block.

At global scale, this adds up fast. Data centers are already consuming record amounts of electricity. As AI agents become part of daily life, the energy cost will only grow — unless we build smarter foundations.

We believe the best agent framework should also be the most efficient one. Not as an afterthought — efficiency should be the starting point.

⚡ What

A framework for agents that stay alive

Sigil is a framework for building AI agents and web applications with Elixir — a language designed for systems that never stop running. Instead of treating each AI interaction as a one-time request, Sigil keeps agents alive as long-running processes.

That means your agent remembers its conversation, keeps its tools ready, and picks up right where it left off — even after a crash. No cold starts. No wasted context. Just continuous, efficient work.

🔧 How

Built on the most battle-tested runtime in the world

Sigil runs on the BEAM — the same virtual machine that powers phone networks carrying billions of calls with 99.9999999% uptime. It was built for exactly this kind of problem: millions of independent processes, running forever, supervised automatically.

When an agent crashes, the system restarts it from its last checkpoint — no human intervention, no data loss. When traffic spikes, new processes spin up in microseconds. When an agent is idle, it consumes almost zero resources.

We combine this with intelligent memory management — context windows that compress and summarize old conversations, so your agents use the fewest tokens possible for each AI call. Less tokens means less compute means less energy.

📊 The Hypothesis

What if agents didn't start from scratch every time?

Most AI frameworks today follow the same pattern: receive a request, spin up a process, load all the context, call the model, return a response, tear everything down. Every single interaction pays the full cost.

Sigil takes a fundamentally different approach. Agents stay alive between interactions. Context is managed intelligently. The runtime is designed for millions of concurrent, idle-cheap processes. Here's how we think this could play out:

Typical approach Sigil on BEAM
Agent lifecycle Spun up per request, torn down after Long-running, stays alive between calls
Context loading Full history reloaded every call Kept in-memory, summarized progressively
Tokens per interaction Grows linearly with conversation length Stays bounded via context budgeting
Idle agent cost Container / thread sitting idle ~2KB suspended BEAM process
Crash recovery Manual retry logic, lost state Auto-restart from last checkpoint
Concurrent agents Hundreds per server (threads / containers) Millions per server (BEAM processes)

🧠 The memory strategy is the key

The biggest cost in running AI agents isn't compute — it's tokens. Every token sent to the model costs money and energy. Most frameworks send the entire conversation history with every request. As conversations grow, so does the cost.

Sigil's memory layer uses progressive summarization and token budgeting to keep context windows small. Old messages are compressed into summaries. The system allocates a fixed token budget across system prompt, history, and response — so you never overflow and never overspend.

For a 50-message conversation, a typical framework might send ~25,000 tokens per call. Sigil's budget system could keep that under ~4,000 tokens — an estimated 80% reduction in token usage per interaction.

These are hypothetical projections based on architectural analysis, not benchmarks. We're building Sigil to test this thesis in practice.

Get Started

Go from zero to a running app in under 5 minutes.

1

Install Elixir

If you don't have Elixir yet, install it with mise (recommended) or asdf:

$ mise install elixir@1.18
$ mise install erlang@27
2

Install the Sigil generator

$ mix archive.install hex sigil_new
3

Create your app

$ mix sigil.new my_app
$ cd my_app
$ mix setup
4

Start the server

$ mix sigil.server

Visit http://localhost:4000. You should see your app's welcome page with a live counter.

🎉 That's it.

You have a running Sigil app with real-time updates, a database, and auth ready to go. Now build something.

Build Something

A progressive tutorial that takes you from a generated app to a personal site with a blog, auth, and AI agents. Each step builds on the last.

1. Your Homepage

Open lib/my_app_web/pages/home_live.ex. This is your homepage — a Sigil Live view. Edit the render/1 function to show your name:

lib/my_app_web/pages/home_live.ex
def render(assigns) do
  """
  <div>
    <h1>Hi, I'm Ada 👋</h1>
    <p>I build things with Sigil.</p>
  </div>
  """
end

Save and refresh. Your server auto-recompiles.

💡 How it works

Sigil.Live views are server-rendered on first load, then updated in real-time via WebSocket. The render/1 function returns HTML as a string. No templates, no JSX — just Elixir string interpolation.

2. Add a Blog

Use AI to generate the code:

$ mix sigil.gen "blog with posts that have a title, body, and published_at date"

This calls your configured LLM, which generates:

  • An Ecto schema (Post) with validations
  • A database migration
  • A context module with CRUD functions
  • Live views for listing and viewing posts

Review the files, confirm, then run the migration:

$ mix ecto.migrate

Add the routes to your router:

lib/my_app/router.ex
defmodule MyApp.Router do
  use Sigil.Router

  live "/", MyAppWeb.HomeLive
  live "/blog", MyAppWeb.BlogLive
  live "/blog/:slug", MyAppWeb.PostLive

  sigil_routes()
end

3. Add Auth

Sigil has auth built in. Create the users table:

$ mix ecto.migrate

Protect any route by adding auth: true:

lib/my_app/router.ex
live "/dashboard", MyAppWeb.DashboardLive, auth: true

Unauthenticated users get redirected to /login. Registration, login, and session management are all built into the framework.

4. Your First Agent

Create a writing assistant that helps draft blog posts:

$ mix sigil.gen.agent WritingAssistant "helps write blog posts in a conversational tone"

This generates two files:

  • lib/my_app/agents/writing_assistant.ex — The agent module
  • lib/my_app_web/pages/writing_assistant_live.ex — A chat UI

Add the route and set your API key:

live "/assistant", MyAppWeb.WritingAssistantLive

# In config/dev.exs:
config :sigil, :gen,
  api_key: System.get_env("ANTHROPIC_API_KEY")
💡 How agents work

Sigil agents are OTP GenServers. They maintain conversation history, manage context windows (so you never overflow the token limit), and checkpoint their state to the database. If they crash, the BEAM supervisor restarts them and they resume from the last checkpoint.

5. Agent Dashboard

Build a dashboard to manage multiple agents. Create a Live view that lists running agents and lets you chat with any of them:

lib/my_app_web/pages/dashboard_live.ex
defmodule MyAppWeb.DashboardLive do
  use Sigil.Live

  def mount(_params, socket) do
    agents = DynamicSupervisor.which_children(Sigil.Agent.Runner)
    {:ok, Sigil.Live.assign(socket, agents: agents)}
  end

  def render(assigns) do
    """
    <h1>Agent Dashboard</h1>
    <p>#{length(assigns.agents)} agents running</p>
    """
  end
end

Add it as a protected route:

live "/dashboard", MyAppWeb.DashboardLive, auth: true

6. Deploy

Your app includes a production Dockerfile. Deploy to Render.com:

  1. Push your code to GitHub
  2. Create a new Web Service on Render
  3. Set environment to Docker
  4. Add environment variables:
DATABASE_URL=postgresql://...
SECRET_KEY_BASE=# run: openssl rand -base64 64
ANTHROPIC_API_KEY=sk-ant-...
PORT=4000

Render will build and deploy your app. That's it — you're live. 🚀

Reference

Sigil is organized into five composable layers, from low-level to high-level.

Layer 5 · Sigil.Live

↑
Layer 4 · Sigil.Agent

↑
Layer 3 · Sigil.Memory

↑
Layer 2 · Sigil.Tool

↑
Layer 1 · Sigil.LLM

Layer 1: Sigil.LLM

Unified interface to AI models. Swap providers without changing code.

Sigil.LLM

Behaviour defining chat/2, stream/2, embed/2

Sigil.LLM.Anthropic

Claude adapter (Sonnet, Opus, Haiku)

Sigil.LLM.OpenAI

GPT adapter (GPT-4o, GPT-4, etc.)

Layer 2: Sigil.Tool

Define actions agents can take, with permissions and timeouts.

Sigil.Tool

Behaviour: name/0, description/0, schema/0, call/2

Sigil.Tools.HTTPRequest

Built-in tool for HTTP API calls via Req

Layer 3: Sigil.Memory

Context window management so agents never overflow token limits.

Sigil.Memory.Context

Manages conversation history with token-aware truncation

Sigil.Memory.Budget

Allocates tokens across system, history, and response

Sigil.Memory.Summarizer

Progressive summarization of old messages

Sigil.Memory.Store

Persistent key-value store per agent

Layer 4: Sigil.Agent

The orchestrator. A GenServer that runs the agent loop.

Sigil.Agent

Core behaviour: init_agent/1. Provides start/2, chat/2

Sigil.Agent.Checkpoint

Persists agent state to database for crash recovery

Sigil.Agent.EventStore

Append-only log of every agent action for replay

Sigil.Agent.Observer

Observability: token usage, decision history, drift detection

Sigil.Agent.Guard

Input sanitization and prompt injection heuristics

Sigil.Agent.Team

Multi-agent coordination with message routing

Layer 5: Sigil.Live

Real-time web views over WebSocket. No JavaScript framework.

Sigil.Live

Behaviour: mount/2, render/1, handle_event/3

Sigil.Router

Routing DSL: live/2, sigil_routes/0, auth protection

Sigil.Layout

HTML layout system with default shell

Sigil.Auth

Register, login, sessions, protected routes
