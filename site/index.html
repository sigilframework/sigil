What is this?

Sigil is two things — a framework and a starter app. The framework gives you composable layers for building AI agents. The starter app shows how they fit together.

🧱

The framework

Add {:sigil, "~> 0.1.4"} to any Elixir project. You get agents, memory management, tool execution, real-time UI, and auth as independent, composable layers. Use one or all of them.

📦

The starter app

Run mix sigil.new my_app and you get a working blog with AI chat, an admin dashboard, and agent tools. It's meant to show how the pieces fit together — poke at it, modify it, rebuild it.

Tip: AI coding tools (Claude Code, Cursor, etc.) work well with Sigil's codebase — the structure is opinionated enough that they can extend it without inventing architecture from scratch.

Why Elixir
for AI agents?

AI agents need concurrency, fault tolerance, and long-lived state. The BEAM was built for all three.

Elixir runs on the BEAM — a runtime built for concurrent, fault-tolerant systems. Here's what that gives you in practice.

Agents are processes.

Each agent is a BEAM process: ~2 KB of memory, isolated heap, preemptive scheduling. Context switches happen in user space at ~20 ns — three orders of magnitude faster than OS threads. WhatsApp ran 2 million connections per server on this model. Phoenix benchmarked the same for WebSockets. No queues, no orchestration layer, no container per agent.

Memory is compressed, not replayed.

Most chat systems resend the full conversation every turn — a 50-message thread can cost 25,000+ tokens per API call. Sigil summarizes older messages progressively, keeping the working context small. The agent retains what matters without paying to re-read the full history.

The UI runs on the same runtime.

Each connected user is a BEAM process. The server renders HTML and pushes diffs over a WebSocket — no React, no build step. Discord serves 11 million concurrent users on Elixir. The same process model that runs your agents also runs your UI.

🧪 The Runtime

A quick intro to the BEAM

Elixir runs on the BEAM — the Erlang VM. It was built by Ericsson in the '80s for telecom switches that can never go down. It turns out many of the problems you hit building AI agents — concurrency, fault tolerance, long-running state — are problems the BEAM solved decades ago.

If you're coming from Python, Node, or Ruby, some of this will feel unfamiliar. That's part of what makes it interesting to explore.

💡 Lightweight processes

Each agent, each chat, each user gets its own process. They're ~2KB each — one server can run millions. Compare that to OS threads (MBs each) or containers.

🔄 Self-healing

If a process crashes, its supervisor restarts it from the last checkpoint. Conversations don't get lost. No manual recovery. The system just keeps going.

⚡ Real-time by default

WebSocket connections are processes too. Streaming chat responses, live UI updates — it's all native. No special infrastructure needed.

🏗️ Battle-tested

The BEAM has 30+ years of production use — from Ericsson telecom switches to WhatsApp and Discord. It's not new or unproven technology.

🌍 An aside on efficiency

AI workloads are energy-intensive. Building on a runtime where processes use a fraction of the resources of containers and threads means less compute per request. I don't know if this matters at small scale, but it's an interesting property to have as things grow.

Build your own (Elixir)

For when you're ready to customize by hand. Install Elixir, generate a Sigil app, and own every line of code.

Step 1 — Install prerequisites

You need three things before using Sigil:

  • • Elixir 1.18+ — includes mix, the build tool used in all commands below
  • • Erlang/OTP 27+ — installed automatically with most Elixir installers
  • • PostgreSQL 14+ — the database (must be running locally)

Install Elixir and Erlang with mise (recommended):

$ mise install elixir@1.18 erlang@27

Or use Homebrew, asdf, or any package manager. For PostgreSQL, see the official downloads.

Step 2 — Install the Sigil generator

$ mix archive.install hex sigil

This installs the mix sigil.new command globally. You only need to do this once.

Step 3 — Create your app and run it

# Generate your app (prompts for your Anthropic API key)
$ mix sigil.new my_app && cd my_app

# Install deps, create database, seed data (PostgreSQL must be running)
$ mix setup

# Start the server
$ mix sigil.server

Visit localhost:4000. You now have a fully customizable Sigil app.

Email: admin@example.com
Password: admin123

You'll need an Anthropic API key for AI chat. The generator will prompt you for it, or you can add it to .env later.

Layer 1: Sigil.LLM

Unified interface to AI models. One API for every provider.

Sigil.LLM

Behaviour: chat/2, stream/2, embed/2

Sigil.LLM.Anthropic

Claude adapter (Sonnet, Opus, Haiku)

Sigil.LLM.OpenAI

GPT adapter (GPT-4o, GPT-4, etc.)

Why this matters: AI models change fast. Last year's best model isn't this year's. Sigil abstracts the provider behind a single behaviour, so you swap from Claude to GPT (or whatever comes next) by changing one line of config. Your business logic never changes. You're never locked to a vendor.

Layer 2: Sigil.Tool

Give your agents the ability to take real actions — not just generate text.

Sigil.Tool

Behaviour: name/0, description/0, schema/0, call/2

Sigil.Tools.HTTPRequest

Built-in tool for external API calls

Why this matters: An agent that can only talk is a chatbot. An agent that can do things — check a calendar, book a meeting, query your database, send an email — is a product. Sigil.Tool is a four-function contract. Implement it once, assign it to any agent from the admin UI. The agent decides when to use it based on the conversation. You define what it can do. The AI decides when.

Layer 3: Sigil.Memory

Context management that keeps costs down and conversations coherent.

Sigil.Memory.Context

Manages conversation history with token-aware truncation

Sigil.Memory.Budget

Allocates tokens across system prompt, history, and response

Sigil.Memory.Summarizer

Progressive summarization — compresses old messages into concise context

Sigil.Memory.Store

Persistent key-value memory per agent

Why this matters: Every AI API charges by the token. Most chat systems resend the entire conversation history with every message — a 50-message chat can burn 25,000 tokens per call. That's wasted money on context the model already processed. Sigil's memory layer compresses older messages into summaries automatically. The agent remembers what happened without paying to re-read every word. Long conversations stay affordable. Context windows never overflow.

Layer 4: Sigil.Agent

The orchestrator. A persistent process that runs the full agent loop.

Sigil.Agent

Core behaviour: init_agent/1. Provides start/2, chat/2

Sigil.Agent.Checkpoint

Persists state to disk — crash recovery without data loss

Sigil.Agent.EventStore

Append-only log of every decision and action

Sigil.Agent.Observer

Token usage tracking, decision history, drift detection

Sigil.Agent.Guard

Input sanitization and prompt injection heuristics

Sigil.Agent.Team

Multi-agent coordination with message routing

Why this matters: Most AI agent frameworks run a script, get a response, and throw everything away. If the process crashes, the conversation is gone. Sigil agents are Erlang/OTP processes — they stay alive between conversations, checkpoint their state to disk, and restart from the last known good state if anything goes wrong. Each agent is ~2KB of memory. One server can run thousands simultaneously. This is the same model that lets WhatsApp handle billions of messages with a small engineering team.

Layer 5: Sigil.Live

Real-time UI over WebSocket. No JavaScript framework. No build step.

Sigil.Live

Behaviour: mount/2, render/1, handle_event/3

Sigil.Router

Routing DSL: live/2, sigil_routes/0, auth guards

Sigil.Layout

HTML layout system with responsive shell

Sigil.Auth

Register, login, sessions, protected routes — built in

Why this matters: Traditional web apps need a frontend framework (React, Vue, Svelte), a build pipeline (Webpack, Vite), API endpoints, and state management. That's thousands of lines of glue code before you ship anything. Sigil.Live renders HTML on the server and pushes updates to the browser over a WebSocket. The client is ~2KB of JavaScript. Pages feel instant. Chat responses stream in real time. You write one language — Elixir — and it handles both the backend and the UI.

Layer 6: Sigil.Auth

Users, sessions, and access control. Not an afterthought.

Sigil.Auth

Registration, login, password hashing, session management

Sigil.Auth.Plugs

Route-level guards: require_auth, require_admin

Why this matters: Auth is the first thing every real product needs and the last thing most frameworks provide. Without it, your AI app is a public demo. With Sigil.Auth, user accounts, admin access, and protected routes work from the first mix sigil.new. No third-party auth service. No monthly fees. No data leaving your server. Your users are yours.

These six layers are designed to work together — but each one is independent. Use the pieces you need.
